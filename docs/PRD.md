# 產品需求文件 (PRD)：語音轉寫與 API 整合系統

## 1. 專案概述

### 背景與動機

在進行使用者訪談、田野調查或團隊會議錄音時，手動將大量語音內容逐字轉寫為文字稿是極為耗時且容易出錯的工作。本模組作為「即時標記與語音辨識錄音工具」的核心後端子系統，負責承接前端上傳的音訊檔案，透過 Flask 路由與 OpenAI Whisper API 進行整合，將語音訊號高精確度地轉化為文字逐字稿，並將轉寫結果與對應的時間軸資料（包含前端錄音時產生的即時標記時間點）同步存入 SQLite 資料庫中，大幅降低人工轉寫的門檻與時間成本。

### 目標用戶

- **訪談員/研究人員**：需要錄製使用者訪談、田野調查，並將語音內容快速轉化為可搜尋、可標記的文字逐字稿。
- **組員 A（錄音操作者）**：負責在前端進行即時錄音與標記重點時間點，錄音完成後觸發上傳與轉寫流程。
- **組員 B（逐字稿整理者）**：在轉寫完成後，利用系統提供的時間軸對齊結果進行逐字稿校對與報告撰寫。

### 核心價值主張

提供一套自動化、高精確度的語音轉文字管線，結合在地化提示詞（台灣國台語混雜場景）與即時標記時間軸對齊功能，讓使用者從「錄音 → 上傳 → 轉寫 → 標記對齊 → 儲存」的完整流程在單一系統內無縫完成。

## 2. 功能需求

### 功能 1：安全的音訊檔案上傳端點

建立專屬的 Flask 上傳路由（Endpoint），接收前端傳來的音訊資料，並在伺服器端進行完整的安全性驗證，包含檔案大小限制檢查、副檔名白名單驗證（僅允許 `.wav`、`.mp3`、`.m4a`、`.webm`、`.ogg` 等合法音訊格式），以及 MIME Type 校驗，防止惡意檔案上傳。

- **使用者故事**：作為一位訪談員，我希望能夠透過前端頁面上傳錄音檔案到後端伺服器，並在上傳時自動檢查檔案合法性，以便確保只有安全的音訊檔案進入系統處理流程。

---

### 功能 2：音訊格式預檢與限制阻擋

在呼叫 Whisper API 前，後端對音訊檔案進行格式預檢，確認檔案是否符合 Whisper API 的接受規格（如：檔案大小上限 25MB、支援的音訊編碼格式等）。若不符合規格，系統立即回傳具體的錯誤提示訊息給前端，避免不必要的 API 呼叫浪費。

- **使用者故事**：作為一位錄音操作者，我希望在上傳格式不支援或檔案過大時，系統能即時告訴我問題出在哪裡，以便我能重新錄製或轉檔後再上傳，而不是等了很久才發現辨識失敗。

---

### 功能 3：在地化 Whisper API 串接與辨識優化

串接 OpenAI Whisper API 進行語音轉文字，並特別在 API 請求的 `prompt` 參數中加入「台灣習慣用語、國台語混雜」的情境提示詞（如：常見的台語語助詞、台灣地名、台灣慣用語等），大幅提升在地化訪談場景的字詞辨識精準度。同時指定 `response_format` 為 `verbose_json` 以取得包含每句起止時間的結構化回傳資料。

- **使用者故事**：作為一位進行台灣在地訪談的研究人員，我希望系統能辨識出國台語混雜的口語內容，並盡可能正確轉寫為文字，以便我後續整理逐字稿時不需要花大量時間修正辨識錯誤。

---

### 功能 4：轉寫結果與即時標記的時間軸對齊

解析 Whisper API 回傳的 JSON 資料，提取每個句子（segment）的 `start` 與 `end` 時間戳，並將其與組員 A 在前端錄音時產生的「即時標記時間點（Markers）」進行時間軸上的對齊。系統會為每個標記找到對應的轉寫段落，並將對齊結果一併寫入 SQLite 資料庫，方便後續查詢與瀏覽。

- **使用者故事**：作為一位逐字稿整理者，我希望能直接看到每個重點標記對應到哪一段轉寫文字與時間區間，以便我能快速定位到訪談的關鍵段落進行校對，而不需要整份逐字稿從頭讀到尾。

---

### 功能 5：非同步處理狀態回報與異常處理

由於語音辨識需要一定的運算時間（依音訊長度可能耗時數秒至數十秒），後端需在整個轉寫流程中提供正確的 HTTP 狀態碼回報（如：`202 Accepted` 表示已接受處理、`200 OK` 表示處理完成），並妥善處理各種異常狀態，包含：API 金鑰無效（`401`）、API 額度不足（`429 Rate Limit`）、網路連線逾時（`504 Gateway Timeout`）、Whisper 服務不可用（`503`）等，回傳語義明確的錯誤訊息供前端顯示。

- **使用者故事**：作為一位系統使用者，我希望在轉寫過程中能清楚知道目前的處理狀態（處理中、完成、失敗），並在發生錯誤時看到具體的錯誤原因與建議操作，以便我知道該等待還是需要重新操作。

## 3. 非功能需求

### 技術限制

- **後端框架**：Python + Flask
- **模板引擎**：Jinja2（用於渲染轉寫結果頁面）
- **資料庫**：SQLite（儲存錄音紀錄、轉寫逐字稿、時間戳與標記資料）
- **外部 API**：OpenAI Whisper API（`whisper-1` 模型）
- **API 金鑰管理**：使用 `.env` 檔案搭配 `python-dotenv` 管理 API 金鑰，避免金鑰洩漏
- **音訊暫存**：上傳的音訊檔案暫存於 `instance/uploads/` 目錄，轉寫完成後可選擇性保留或清除

### 效能考量

- 音訊檔案大小上限設為 **25MB**（配合 Whisper API 限制）
- 單次轉寫請求需設定合理的 API 呼叫逾時時間（建議 **120 秒**）
- 前端上傳需顯示進度指示，後端回傳適當的狀態碼以支援前端輪詢（Polling）或非同步通知

### 安全考量

- 對上傳檔案進行副檔名白名單過濾與 MIME Type 驗證，防範惡意檔案上傳
- API 金鑰不可寫死在程式碼中，需透過環境變數讀取
- 使用 Flask 內建的安全機制防範 CSRF 攻擊（表單操作時）
- 防範 SQL Injection：使用參數化查詢存取 SQLite
- Jinja2 預設的自動轉義功能防護 XSS

## 4. MVP 範圍

### Must Have（必備功能 — 初期實作）

- 音訊檔案上傳路由，含檔案大小與副檔名安全驗證
- 音訊格式預檢（Whisper API 規格檢查），不合規則回傳錯誤提示
- 串接 OpenAI Whisper API 執行語音轉文字，含在地化 prompt 提示詞
- 解析 API 回傳結果，儲存逐字稿文字與每句段落的時間戳至 SQLite
- 提供正確的 HTTP 狀態碼回報與常見 API 錯誤處理（逾時、額度不足等）

### Should Have（應有功能 — 中期實作）

- 轉寫結果與前端即時標記（Markers）的時間軸對齊邏輯
- 轉寫結果瀏覽頁面（以 Jinja2 渲染，顯示逐字稿、時間軸與標記）
- 轉寫歷史紀錄列表（查看過去上傳的所有錄音與轉寫結果）

### Nice to Have（加分功能 — 未來擴充）

- 音訊檔案自動格式轉換（透過 FFmpeg 將不支援的格式轉為 Whisper 可接受格式）
- 轉寫結果匯出為文字檔（`.txt`）或 SRT 字幕檔（`.srt`）
- 前端即時顯示轉寫進度（搭配 WebSocket 或 Server-Sent Events）
- 支援多語言辨識模式切換（中文 / 英文 / 自動偵測）
